# <a name="how-exchange-online-uses-tls-to-secure-email-connections-in-office-365"></a>Использование протокола TLS службой Exchange Online для защиты электронной почты в Office 365

Узнайте, как Exchange Online и Office 365 использовать безопасности TLS (Transport Layer) и вперед безопасной пересылки (FS) для защиты подключений по электронной почте. Также предоставляет сведения о сертификат, выданный корпорацией Майкрософт для Exchange Online.
  
## <a name="tls-basics-for-office-365-and-exchange-online"></a>Основные сведения о протоколе TLS для Office 365 и Exchange Online

Безопасность TLS (Transport Layer) и протокол SSL, поставляемый перед TLS, являются протоколов шифрования, которые безопасный обмен данными по сети с помощью сертификатов безопасности для шифрования подключения между компьютерами. Протокол TLS заменило Secure Sockets Layer (SSL) и часто называется SSL 3.1. Для Exchange Online мы используем TLS для шифрования подключения между наших серверов Exchange и подключения между наших серверов Exchange и других серверов, например, локальные серверы Exchange или получателей почтовых серверов. Зашифрованное соединение через зашифрованный канал отправляется все данные, передаваемые через это подключение. Тем не менее если переслать сообщение, которое было отправлено через подключение зашифрованных с использованием TLS, это сообщение не обязательно шифруются. Это так, как в простой термины TLS не шифрование сообщения, просто подключения.
  
Чтобы зашифровать сообщение, необходимо использовать технологию, которая шифрует содержимое сообщения, например шифрование сообщений Office. Сведения о вариантах шифрования в Office 365 см. в статьях [Email encryption in Office 365](email-encryption.md) и [Office 365 Message Encryption (OME)](ome.md). 
  
Рекомендуется использовать TLS в ситуациях, в которых вы собираетесь установить безопасный канал между Office 365 и локальной организации или другой организации, такие как партнера. Exchange Online всегда сначала пытается использовать TLS для защиты электронной почты, но всегда невозможно, если не удается другой стороны безопасности TLS. Продолжайте читать, чтобы узнать, как можно защитить всю почту на локальные серверы или важные партнеров с помощью *соединителей*. 
  
## <a name="how-exchange-online-uses-tls-between-exchange-online-customers"></a>Использование протокола TLS службой Exchange Online для защиты данных, передаваемых между пользователями этой службы

Серверы Exchange Online всегда шифруют соединения с другими серверами Exchange Online в наших центрах обработки данных с помощью протокола TLS 1.2. Когда вы отправляете сообщение пользователю, находящемуся в вашей организации Office 365, оно автоматически передается через соединение, зашифрованное с помощью протокола TLS. Кроме того, все сообщения, отправляемые другим пользователям Office 365, передаются через соединения, зашифрованные с помощью TLS и защищенные с помощью функции безопасной пересылки.
  
## <a name="how-office-365-uses-tls-between-office-365-and-external-trusted-partners"></a>Как Office 365 использует протокол TLS между внешним, доверенных партнеров и Office 365

По умолчанию Exchange Online всегда использует гибкий протокол TLS. Это означает, что Exchange Online всегда пытается сначала шифрования соединений с версией самый безопасный протокол TLS, а затем идет вниз по списку из шифры TLS, пока не находит на котором можно принять обе стороны. Если вы настроили Exchange Online, чтобы убедиться, что сообщения, получателю только через безопасного подключения, затем по умолчанию сообщения будут отправляться незашифрованных Если организация получателя не поддерживает шифрование TLS. Гибкий протокол TLS является достаточным для большинства компаний. Тем не менее для бизнеса, требования соответствия, такие как медицинских, банка или государственных организаций, можно настроить Exchange Online, чтобы сделать обязательным наличие или принудительный протокол TLS. Сведения содержатся в разделе [Настройка потока обработки почты с помощью соединителей в Office 365](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx).
  
Если вы решили установить TLS между вашей организацией и организацией доверенным партнером, Exchange Online можно использовать принудительный протокол TLS для создания надежного каналы связи. Принудительный протокол TLS требует вашей партнерской организации проходить проверку подлинности в Exchange Online с помощью сертификата безопасности для отправки почты. Ваш партнер должен управлять свои собственные сертификаты для этого. В Exchange Online мы используем соединителей для защиты отправляемых сообщений от несанкционированного доступа перед прибытия у поставщика электронной почты получателя. Сведения на с помощью соединителей для настройки потока обработки почты можно [настроить поток обработки почты с помощью соединителей в Office 365](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx).
  
## <a name="tls-and-hybrid-exchange-server-deployments"></a>Протокол TLS и гибридные развертывания Exchange Server

При управлении гибридного развертывания Exchange потребностей в локальной Exchange server проходить проверку подлинности в Office 365 с помощью сертификата безопасности для отправки почты получателям, почтовые ящики находятся только в Office 365. В результате необходимо управление сертификатами безопасности для локальных серверов Exchange. Необходимо также безопасно хранить и поддерживать эти сертификаты сервера. Дополнительные сведения об управлении сертификатами в гибридных развертываниях можно [требования к сертификатам для гибридных развертываний](https://technet.microsoft.com/library/hh563848%28v=exchg.150%29.aspx).
  
## <a name="how-to-set-up-forced-tls-for-exchange-online-in-office-365"></a>Как настроить принудительное использование TLS для Exchange Online в Office 365

Для клиентов Exchange Online чтобы принудительный протокол TLS для работы для обеспечения безопасности, все отправки и получения электронной почты, необходимо настроить несколько соединителей, которому требуется TLS. Вам понадобится один соединитель для электронной почты, отправляемых на ваши почтовые ящики пользователей и другой соединитель для электронной почты, отправляемых из почтового ящика пользователя. Создавайте их в центре администрирования Exchange в Office 365. Сведения содержатся в разделе [Настройка потока обработки почты с помощью соединителей в Office 365](https://technet.microsoft.com/library/ms.exch.eac.connectorselection%28v=exchg.150%29.aspx).

## <a name="prepare-for-the-new-exchange-online-certificate"></a>Подготовка для создания нового сертификата Exchange Online

Новый сертификат выдан различных сертификации (ЦС) из предыдущей сертификат, используемый Exchange Online. В результате может потребоваться выполнить некоторые действия, чтобы использовать новый сертификат.

Новый сертификат требуется подключение к конечным точкам нового ЦС в составе проверке сертификата. В противном случае можно привести негативного влияния поток почты. Если ваш почтовый сервер с помощью брандмауэров, которые позволяют подключать к определенной назначения, следует проверить, если сервер является возможность проверить новый сертификат почтовые серверы защиты. Чтобы убедиться в том, что сервер можно использовать новый сертификат, выполните следующие действия:

1. Подключение к локальному серверу Exchange с помощью Windows PowerShell, а затем выполните следующую команду:  
  `certutil -URL http://crl.globalsign.com/gsorganizationvalsha2g3.crl`
2. В появившемся окне выберите команду **извлечь**.
3. По завершении проверку служебной программы возвращает состояние. Если состояние отображает **кнопку ОК**, ваш почтовый сервер успешно можно проверить новый сертификат. Если нет, необходимо определить причину подключений к ошибке. Скорее всего вам потребуется обновить параметры брандмауэра. Полный список конечных точек, которые должны быть доступны следующие:
    - OCSP.GlobalSign.com
     - CRL.GlobalSign.com
     - Secure.GlobalSign.com   

Как правило получение обновлений для вашей корневые сертификаты автоматически через Центр обновления Windows. Однако некоторые развертываний имеют обеспечения дополнительного уровня безопасности на месте, которое не позволяет эти обновления происходит автоматически. В развертываниях эти заблокированные, где центра обновления Windows не удается обновить автоматически корневые сертификаты необходимо установить правильный корневого Центра сертификации, выполнив следующие действия:
1.  Подключение к локальному серверу Exchange с помощью Windows PowerShell, а затем выполните следующую команду:  
  `certmgr.msc`
2. В разделе **Доверенные корневые сертификаты центра сертификации и**убедитесь, что указано новый сертификат.
