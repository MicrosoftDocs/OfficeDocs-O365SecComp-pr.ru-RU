# <a name="set-up-compliance-boundaries-for-ediscovery-investigations-in-office-365"></a>Настройка границ соответствия для расследований дел обнаружения электронных данных в Office 365

Границы соответствия требованиям создают логические границы в организации Office 365, которые управляют расположениями содержимого пользователя (например, почтовыми ящиками, сайтами SharePoint и учетными записями OneDrive), которые руководители обнаружения электронных данных могут выполнять поиск. Кроме того, границы соответствия требованиям контролируют, кто может получить доступ к делам eDiscovery, используемым для управления юридическим, человеческим персоналом или другим расследованиям в Организации. Потребность в границах соответствия требованиям часто требуется для многопользовательских компаний, которые должны учитывать географические платы и нормативы, а также для государственных учреждений, которые часто делятся на различные учреждения. В Office 365 ограничения соответствия требованиям обеспечивают соблюдение этих требований при выполнении поиска контента и управлении расследованиями с использованием eDiscovery.
  
Мы будем использовать пример, приведенный на следующем рисунке, чтобы объяснить, как работают границы соответствия требованиям.
  
![Границы соответствия требованиям состоят из фильтров разрешений на поиск, которые контролируют доступ к учреждениям и группам ролей администраторов, которые контролируют доступ к случаям Едисоковери](media/5c206cc8-a6eb-4d6b-a3a5-21e158791f9a.png)
  
В этом примере Contoso LTD — это организация Office 365, состоящая из двух дочерних компаний, четвертого кофе и Coho Winery. Для бизнеса необходимо, чтобы диспетчеры обнаружения электронных данных и исследования могли выполнять поиск только в почтовых ящиках Exchange, учетных записях OneDrive и сайтах SharePoint в их компетентном органе. Кроме того, менеджеры обнаружения и обнаружения электронных данных могут видеть только случаи обнаружения электронных данных в агентствох, и они могут получать доступ только к тем случаям, в которых они являются участниками. Вот как границы соответствия требованиям соответствуют этим требованиям.
  
- Функции фильтрации разрешений поиска в поиске контента управляют расположением контента, в котором руководители обнаружения и обнаружения электронных данных могут выполнять поиск. Это означает, что менеджеры по обнаружению электронных данных и средства исследования в четвертом агентстве могут искать только расположения контента в четвертом представительстве кофе. Одно и то же ограничение распространяется на дочернюю компанию Coho Winery.
    
    Группы ролей контролируют, кто может видеть случаи обнаружения электронных данных в центре &amp; безопасности и соответствия требованиям Office 365. Это означает, что менеджеры по обнаружению электронных данных и изучению могут видеть только случаи обнаружения электронных данных в их компетентности.
    
- Группы ролей также контролируют, кто может назначать участников для случая обнаружения электронных данных. Это означает, что менеджеры обнаружения электронных данных и средства исследования могут назначать участников только тем случаям, в которых они сами являются.

## <a name="step-1-identify-a-user-attribute-to-define-your-agencies"></a>Шаг 1: определение атрибута пользователя для определения ваших учреждений

Первый шаг — выбрать атрибут Azure Active Directory, который будет использоваться для определения ваших учреждений. Этот атрибут будет использоваться для создания фильтра разрешений поиска, ограничивающего диспетчером обнаружения электронных данных поиском только расположений контента пользователей, которым назначено определенное значение для этого атрибута. Например, предположим, что компания Contoso решает использовать атрибут **Department** . Значение этого атрибута для пользователей в четвертом представительстве кофе будет равно `FourthCoffee` , а значение для пользователей в Coho Wineryной дочерней `CohoWinery`компании будет. На шаге 4 вы будете использовать эту `attribute:value` комбинацию (например, *Department: FourthCoffee* ), чтобы ограничить места для пользователей, которые руководители обнаружения электронных данных могут выполнять поиск. 
  
Ниже приведен список атрибутов пользователей Azure Active Directory, которые можно использовать для ограничения соответствия требованиям:
  
- Company
    
- CustomAttribute1 — CustomAttribute15
    
- Отдел
    
- Кабинет
    
Хотя доступны дополнительные атрибуты пользователей, особенно для почтовых ящиков Exchange, перечисленные выше атрибуты являются единственными, которые в настоящее время поддерживаются в OneDrive.
  
## <a name="step-2-file-a-request-with-microsoft-support-to-synchronize-the-user-attribute-to-onedrive-accounts"></a>Шаг 2: Отправьте запрос в службу поддержки Майкрософт, чтобы синхронизировать атрибут пользователя с учетными записями OneDrive

Следующий шаг — файл запроса со службой поддержки Майкрософт для синхронизации атрибута Azure Active Directory, выбранного на шаге 1, со всеми учетными записями OneDrive в Организации. После выполнения этой синхронизации атрибут (и его значение), выбранный на шаге 1, будет сопоставлен с скрытым управляемым свойством в SharePoint с именем `ComplianceAttribute`. Этот атрибут используется для создания фильтра разрешений поиска для OneDrive в действии 4.
  
Если вы отправите запрос в службу поддержки Майкрософт, включите следующие сведения:
  
- Доменное имя по умолчанию для организации Office 365
    
- Имя атрибута Azure Active Directory (из шага 1)
    
- Следующий заголовок или описание назначения запроса поддержки: "включить синхронизацию OneDrive для бизнеса с Azure Active Directory для фильтров безопасности соответствия". Это поможет отправить запрос в службу проектирования обнаружения электронных данных Office 365, которая будет реализовывать запрос.
    
После внесения изменений и синхронизации атрибута с OneDrive служба поддержки Майкрософт отправляет вам номер сборки, в котором было внесено изменение, и оценочная Дата развертывания. Обратите внимание, что после того как вы отправите запрос на поддержку, процесс развертывания обычно занимает 4-6 недель.
  
 **Важно!** Перед развертыванием изменения можно выполнить шаг 3 до этапа 5. Но при выполнении поиска контента документы с сайтов OneDrive, указанные в фильтре разрешений поиска, не будут возвращаться, пока не будет развернуто изменение. 
  
## <a name="step-3-create-a-role-group-for-each-agency"></a>Шаг 3: создание группы ролей для каждого агентства

Следующий шаг состоит в создании групп ролей в центре безопасности &amp; Office 365, который будет соответствовать вашим учреждениям. Рекомендуется создать новую группу ролей, скопировав встроенную группу "руководители обнаружения электронных данных", добавив соответствующих участников и удалив роли, которые могут быть неприменимы к вашим потребностям. Дополнительные сведения о ролях, связанных с обнаружением электронных данных, приведены [в разделе Назначение разрешений обнаружения &amp; электронных данных в центре безопасности и соответствия требованиям Office 365](assign-ediscovery-permissions.md).
  
Чтобы создать группы ролей, перейдите на страницу **разрешения** в центре обеспечения безопасности &amp; и создайте группу ролей для каждой команды в каждом агентстве, которая будет использовать границы соответствия требованиям и случаи обнаружения электронных данных для управления расследованиями. 
  
С помощью сценария "границы соответствия требованиям Contoso" необходимо создать четыре группы ролей и добавить в каждую из них соответствующие элементы.
  
- Четвертые менеджеры обнаружения электронных данных в кафе
    
- Четвертые исследования
    
- Менеджеры обнаружения электронных данных Coho Winery
    
- Coho Winery
    

  
## <a name="step-4-create-a-search-permissions-filter-to-enforce-the-compliance-boundary"></a>Шаг 4: Создание фильтра разрешений поиска для применения границы соответствия требованиям
<a name="step4"> </a>

После создания групп ролей для каждого агентства следующим шагом является создание фильтров разрешений поиска, которые связывают каждую группу ролей с определенным агентством, и определяет саму границу соответствия. Необходимо создать один фильтр разрешений поиска для каждого агентства. Дополнительные сведения о создании фильтров разрешений безопасности приведены в разделе [Настройка фильтрации разрешений для поиска контента](permissions-filtering-for-content-search.md).
  
Ниже приведен синтаксис, который используется для создания фильтра разрешений поиска, используемого для границ соответствия требованиям.

```
New-ComplianceSecurityFilter -FilterName <name of filter> -Users <role groups> -Filters "Mailbox_<Compliance attribute from Step 1>  -eq '<AttributeVale> '", "Site_ComplianceAttribute  -eq <AttributeValue>' -or Site_Path -like <SharePointURL> *'" -Action <Action >
```
  
Ниже приведено описание каждого параметра в команде.
  
-  `FilterName`— Задает имя фильтра. Используйте имя, описывающее или идентифицирующее агентство, в котором будет использоваться фильтр. 
    
-  `Users`— Указывает пользователей или группы, которые получают этот фильтр, примененный к выполняемым действиям при поиске контента. Для границ соответствия этот параметр указывает группы ролей (которые вы создали на шаге 3) в агентство, для которого создается фильтр. Примечание Это многозначный параметр, чтобы можно было включить одну или несколько групп ролей, разделив их запятыми. 
    
-  `Filters`— Задает критерии поиска для фильтра. Для границ соответствия необходимо определить следующие фильтры. Каждый из них применяется к расположению пользовательского содержимого. 
    
  -  `Mailbox`— Указывает почтовые ящики, которые могут выполнять `Users` Поиск групп ролей, определенных в параметре. Для границ соответствия *комплианцеаттрибуте* — это тот же атрибут, который был определен в шаге 1, и *AttributeValue* указывает агентство. Этот фильтр разрешает членам группы ролей выполнять поиск только почтовых ящиков в определенном агентство; Пример: `"Mailbox_Department -eq 'FourthCoffee'"` . 
    
  -  `Site`— Указывает учетные записи OneDrive, которые могут выполнять поиск групп `Users` ролей, определенных в параметре. Для фильтра OneDrive используйте собственно строку `ComplianceAttribute`; Он будет сопоставлен с тем же атрибутом, который был определен в действии 1 и синхронизирован с учетными записями OneDrive в результате запроса на получение поддержки, отправленного в шаге 2.  *AttributeValue* указывает агентство. Этот фильтр разрешает членам группы ролей выполнять поиск только в учетных записях OneDrive в определенном агентство; Пример: `"Site_ComplianceAttribute -eq 'FourthCoffee'"`.
    
  -  `Site_Path`— Указывает сайты SharePoint, которые могут выполнять поиск в группах ролей `Users` , определенных в параметре. *SharePointURL* указывает сайты в агентстве, которые могут выполнять поиск участниками группы ролей; Например`"Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'"`
    
-  `Action`-Указывает тип действия поиска соответствия, к которому применяется фильтр. Например, фильтр `-Action Search` применяется только в том случае, когда члены групп ролей, определенные в `Users` параметре, выполняют поиск контента. В этом случае фильтр не будет применен при экспорте результатов поиска. Для обеспечения соответствия требованиям `-Action All` примените фильтр ко всем действиям поиска. 
    
    Список действий по поиску контента представлен в разделе "New – ComplianceSecurityFilter" в разделе [Настройка фильтрации разрешений для поиска контента](permissions-filtering-for-content-search.md#new-compliancesecurityfilter).
    
Ниже приведены примеры двух фильтров разрешений поиска, которые будут созданы для поддержки сценария "границы соответствия требованиям Contoso".
  
 **Четвертый кофе**

```
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_ComplianceAttribute -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL
```
   
 **Coho Winery**

```
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_ComplianceAttribute -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL
```

## <a name="step-5-create-an-ediscovery-case-for-an-intra-agency-investigations"></a>Шаг 5: создание дела обнаружения электронных данных для расследования внутри агентств

Последним шагом является создание нового дела обнаружения электронных данных в центре безопасности &amp; и Добавление группы ролей, созданной на шаге 3, в качестве участника этого случая. Это приводит к двум важным характеристикам использования границ соответствия требованиям:
  
- Только участники группы ролей, добавленные в обращение, смогут просматривать и получать доступ к делам в центре безопасности &amp; и соответствия требованиям. Например, если четвертая группа ролей в кафе является единственным участником, то участники четвертой группы ролей "кафе eDiscovery" (или члены любой другой группы ролей) не смогут просматривать обращение и получать к нему доступ.
    
- Когда участник группы ролей, назначенный для обращения, выполняет поиск, связанный с этим обращением, он будет иметь возможность выполнять поиск в расположениях контента в рамках своего агентства (которое определяется фильтром разрешений поиска, созданным на шаге 4).


Для создания нового дела и назначения участников:
    
1. Перейдите на страницу **Обнаружение электронных** данных в центре &amp; обеспечения безопасности и создайте новое обращение. 
    
2. В списке вариантов обнаружения электронных данных щелкните имя только что созданного вами случая.
    
3. На всплывающей странице **Управление этим обращением** в разделе Управление **группами ролей**нажмите ![кнопку Добавить](media/8ee52980-254b-440b-99a2-18d068de62d3.gif) значок **Добавить**.
    
    ![Добавление группы ролей в качестве участника дела eDiscovery](media/f8b4b557-01b9-4388-85be-b5b5ab7c5629.png)
  
4. В списке групп ролей выберите одну из групп ролей, созданных на шаге 3, и нажмите кнопку **Добавить**.
    
5. В всплывающем окне **Управление этим обращением** нажмите кнопку **сохранить** , чтобы сохранить изменения. 

## <a name="compliance-boundary-limitations"></a>Ограничения на границы соответствия требованиям

При управлении делами обнаружения электронных данных и расследования на соответствие требованиям необходимо учитывать следующие ограничения.
  
- При создании и выполнении поиска контента можно выбирать расположения контента вне вашего агентства. Однако из-за фильтра разрешений на поиск содержимое из этих расположений не будет включено в результаты поиска.
    
- Границы соответствия требованиям не применяются к удержаниям в случаях обнаружения электронных данных. Это означает, что менеджер обнаружения электронных данных в одном агентстве может помещать пользователя в другое агентство по удержанию. Однако граница соответствия будет применена, если диспетчер обнаружения электронных данных просматривает расположения содержимого пользователя, который был включен в удержание. Это означает, что диспетчер обнаружения электронных данных не сможет выполнять поиск в расположении контента пользователя, несмотря на то, что они смогли поместить пользователя на удержание.
    
    Кроме того, статистика хранения будет применяться только к расположениям контента в агентстве.
    
- Фильтры разрешений на поиск не применяются к общедоступным папкам Exchange.

## <a name="searching-and-exporting-content-in-multi-geo-environments"></a>Поиск и экспорт контента в средах с поддержкой нескольких регионов

Фильтры разрешений на поиск также позволяют контролировать место маршрутизации контента для экспорта и то, какой центр обработки данных может выполнять поиск при поиске на сайтах SharePoint и в учетных записях OneDrive в [среде SharePoint с поддержкой нескольких регионов](https://go.microsoft.com/fwlink/?linkid=860840):
  
- Экспорт результатов поиска из определенного центра обработки данных. Это означает, что вы можете указать расположение центра обработки данных, из которого будут экспортироваться результаты поиска.
    
- Маршрутизация Поиск сайтов SharePoint и учетных записей OneDrive в центре обработки данных. Это означает, что вы можете указать расположение центра обработки данных, в котором будет выполняться поиск.
    
Используйте параметр **Region** для командлетов **New/ComplianceSecurityFilter** или **Set – ComplianceSecurityFilter** , чтобы создать или изменить центр обработки данных, через который будет маршрутизироваться экспорт.
  
|**Значение параметра**|**Расположение центра обработки данных**|
|:-----|:-----|
|NAM  <br/> |Северная Америка (фактические центры обработки данных в США)  <br/> |
|EUR  <br/> |Europe  <br/> |
|APC  <br/> |Азиатско — Тихоокеанский регион  <br/> |
|CAN <br/> |Канада
   
Аналогичным образом можно использовать следующие значения для значений параметров **Region** , чтобы управлять центром обработки данных, в котором будут выполняться поиски контента при поиске в расположениях SharePoint и OneDrive. Обратите внимание, что в приведенной ниже таблице также показано, через какие экспортные центры данных будут маршрутизироваться. 
  
|**Значение параметра**|**Расположения маршрутизации центра обработки данных для экспорта**|
|:-----|:-----|
|NAM  <br/> |Россия  <br/> |
|EUR  <br/> |Europe  <br/> |
|APC  <br/> |Азиатско — Тихоокеанский регион  <br/> |
|CAN  <br/> |Россия  <br/> |
|AUS  <br/> |Азиатско — Тихоокеанский регион  <br/> |
|KOR  <br/> |Центр обработки данных по умолчанию в Организации  <br/> |
|GBR  <br/> |Europe  <br/> |
|JPN  <br/> |Азиатско — Тихоокеанский регион  <br/> |
|IND  <br/> |Азиатско — Тихоокеанский регион  <br/> |
|Лам  <br/> |Россия  <br/> |
   
 **Примечание:** Если не указать параметр Region для фильтра разрешений поиска, будет выполняться поиск по умолчанию для области SharePoint "Организации", а затем результаты поиска будут экспортированы в ближайший центр обработки данных. 
  
Ниже приведены примеры использования параметра **– Region** при создании фильтров разрешений поиска для границ соответствия требованиям. Предполагается, что Четвертый филиал компании находится в Северной Америке, а Coho Winery — в Европе. 
  
```
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_Department -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL -Region NAM
```

```
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_Department -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL -Region EUR
```
   
При поиске и экспорте контента в средах с поддержкой нескольких регионов учитывайте следующие моменты.
  
- Параметр **Region** не управляет поиском в почтовых ящиках Exchange; при поиске в почтовых ящиках будут выполняться поиск во всех центрах обработки данных. Чтобы ограничить область, в которой можно выполнять поиск почтовых ящиков **** Exchange, используйте параметр Filters при создании или изменении фильтра разрешений поиска. 
    
- Если диспетчер обнаружения электронных данных необходим для поиска в нескольких регионах SharePoint, необходимо создать для этого диспетчера обнаружения электронных данных другую учетную запись, которую можно использовать в фильтре разрешений поиска, чтобы указать альтернативную область, в которой Размещаются сайты SharePoint или учетные записи OneDrive.
    
- При поиске контента в SharePoint и OneDrive параметр **Region** указывает, что выполняется поиск в основном или вспомогательном расположении, в котором диспетчер обнаружения электронных данных будет проводить расследования обнаружения электронных данных. Если диспетчер обнаружения электронных данных выполняет поиск в сайтах SharePoint и OneDrive за пределами области, указанной в фильтре разрешений поиска, результаты поиска не возвращаются. 
    
- При экспорте результатов поиска содержимое из всех расположений контента (включая Exchange, Skype для бизнеса, SharePoint, OneDrive и другие службы Office 365, которые можно искать с помощью средства поиска контента), будет отправлено в расположение хранилища Azure в центр обработки данных, заданный параметром **Region** . Это помогает организациям оставаться в пределах соответствия требованиям, не позволяя экспортировать контент через контролируемые границы. Если в фильтре разрешений поиска область не указана, содержимое будет отправлено в область по умолчанию для Организации. 
    
- Вы можете изменить существующий фильтр разрешений поиска, чтобы добавить или изменить область, выполнив следующую команду:

    ```
    Set-ComplianceSecurityFilter -FilterName <Filter name>  -Region <Region>
    ```
 
## <a name="frequently-asked-questions"></a>Вопросы и ответы

 **Кто может создавать фильтры разрешений поиска и управлять ими (с помощью командлетов New-ComplianceSecurityFilter и Set-ComplianceSecurityFilter)?**
  
Для создания, просмотра и изменения фильтров разрешений поиска необходимо быть членом группы ролей "Управление организацией" в центре безопасности &amp; и соответствия требованиям.
  
 **Если диспетчер обнаружения электронных данных назначен нескольким группам ролей, которые охватывают несколько агентств, как они выполняют поиск контента в одном органе или на другом?**
  
Диспетчер обнаружения электронных данных может добавлять параметры в поисковый запрос, который ограничит поиск определенным агентством. Например, если в Организации указано свойство **CustomAttribute10** для различения учреждений, они могут добавлять в поисковый запрос следующий код для поиска в почтовых ящиках и в учетных записях `CustomAttribute10:<value> AND Site_ComplianceAttribute:<value>`OneDrive в определенном учреждении:.
  
 **Что произойдет, если изменяется значение атрибута, используемого в качестве атрибута соответствия в фильтре разрешений поиска?**
  
Для применения фильтра разрешений поиска к границе соответствия при изменении значения атрибута, используемого в фильтре, требуется 3 дня. Например, в сценарии Contoso Предположим, что пользователь в четвертом агентстве кофе переносится в агентство по Coho Winery. В результате значение атрибута **Department** объекта User изменяется с *FourthCoffee* на *коховинери* . В этом случае четвертые обнаружения электронных данных и инвесторов будут получать результаты поиска для этого пользователя в течение 3 дней после изменения атрибута. Аналогичным образом может потребоваться до 3 дней, прежде чем Coho Winery обнаружения электронных данных и исследования получат результаты поиска для пользователя. 
  
 **Может в диспетчере обнаружения электронных данных просматриваться содержимое на основе двух разных ограничений соответствия требованиям?**
  
Нет. Это можно сделать, добавив пользователя в группы ролей, которые имеют видимость для обоих учреждений.
  
 **Работают ли фильтры разрешений поиска с удержанием электронных данных, политиками хранения Office 365 или DLP?**
  
Нет, не на данный момент
  
 **Если указать регион для управления экспортом контента, но у меня нет организации SharePoint в этом регионе, можно ли по-прежнему выполнять поиск в SharePoint?**
  
Если область, указанная в фильтре разрешений поиска, отсутствует в Организации, будет выполняться поиск по области по умолчанию.
  
 **Каково максимальное число фильтров разрешений поиска, которые могут быть созданы в Организации?**
  
Количество фильтров разрешений поиска, которые могут быть созданы в Организации, не ограничено. Однако производительность поиска окажет влияние на более 100 фильтров разрешений поиска. Чтобы максимально уменьшить количество фильтров разрешений поиска в Организации, создайте фильтры, которые объединяют правила для Exchange, SharePoint и OneDrive в один фильтр разрешений поиска везде, где это возможно.
