# <a name="set-up-compliance-boundaries-for-ediscovery-investigations-in-office-365"></a>Настройка границ соответствия для расследований дел обнаружения электронных данных в Office 365

Границы соответствия Создание логических границ в пределах организации Office 365, которые управляют пользователя расположения контента (например, почтовые ящики, сайты SharePoint и учетные записи OneDrive), которые могут выполнять поиск диспетчеры обнаружения электронных данных. Кроме того соответствия границы контролировать, кто может получить доступ к вариантах eDiscovery, используемый для управления юридическим, отдела кадров и других расследований вашей организации. Необходимость границ соответствия часто это необходимо для нескольких nations организаций, которые должны использовать географических атаки и правилам и для государственных организаций, разделенные на разных органов часто. В Office 365, соответствия границы помогут вам провести поиск и управление расследований с вариантах eDiscovery содержимого эти требования при выполнении.
  
Мы используем пример на следующем рисунке объяснить, как работают границы соответствия требованиям.
  
![Соответствие требованиям границы состоят из поисковых фильтров разрешения групп, что управление доступом к службам и роль администратора, управляющих доступом к eDisocovery случаев](media/5c206cc8-a6eb-4d6b-a3a5-21e158791f9a.png)
  
В этом примере Contoso LTD — это организации Office 365, которая состоит из двух дочерних подразделений, Fourth Coffee и Coho Winery. Бизнес-требованиям, что руководители обнаружения электронных данных и следствие можно только поиска почтовых ящиков Exchange, учетных записей OneDrive и сайты SharePoint в их бюро. Кроме того, диспетчеры обнаружения электронных данных и следователям доступны только вариантах eDiscovery в в их бюро и они доступны только случаи, которые они должна быть членом. Вот как границы соответствия соответствовать следующим требованиям.
  
- Фильтрация в элементах управления контентом поиска расположения контента, которые может выполнять поиск eDiscovery руководителям и следствие разрешения поиска. Это означает, что диспетчеры обнаружения электронных данных и следователям в бюро Fourth Coffee можно только поиск расположения контента в дочернем Fourth Coffee. Это же ограничение применяется к дочерним подразделением Coho Winery.
    
    Роль групп контролировать, кто может просматривать вариантах eDiscovery безопасности Office 365 &amp; центре соответствия требованиям. Это означает, что диспетчеры обнаружения электронных данных и следователям доступны только вариантах eDiscovery в их бюро.
    
- Группы ролей также контролировать, кто может назначать члены к обращению eDiscovery. Это означает, что диспетчеры обнаружения электронных данных и следователям могут быть назначены только члены для случаев, которые они сами, являющийся членом.

## <a name="step-1-identify-a-user-attribute-to-define-your-agencies"></a>Этап 1: Определение атрибут пользователя для определения вашего органов (en)

Первым этапом является, выберите атрибут Azure Active Directory для использования, который будет определить вашей органов. Этот атрибут будет использоваться для создания фильтра разрешения поиска, которая ограничивает eDiscovery диспетчера для поиска только размещения содержимого, пользователей, которым назначена определенного значения этого атрибута. Например предположим, компания Contoso решила использовать атрибут **отдела** . Значение этого атрибута для пользователей в дочерней компании Fourth Coffee будет `FourthCoffee` и значение для пользователей в дочерней Coho Winery будет `CohoWinery`. На шаге 4, он используется в `attribute:value` пары (например, *Отдел: FourthCoffee* ) для ограничения пользователя содержимого расположения, которые могут выполнять поиск диспетчеры обнаружения электронных данных. 
  
Ниже приведен список атрибутов пользователя Azure Active Directory, которые можно использовать для ограничения соответствия требованиям:
  
- Company
    
- CountryCode
    
- CustomAttribute1 — CustomAttribute15
    
- Отдел
    
- Кабинет
    
Хотя особенно для почтовых ящиков Exchange доступны дополнительные атрибуты пользователя, перечисленных выше атрибуты являются единственными людьми в настоящее время поддерживается OneDrive.
  
## <a name="step-2-file-a-request-with-microsoft-support-to-synchronize-the-user-attribute-to-onedrive-accounts"></a>Шаг 2: Файл запроса с помощью службы поддержки Майкрософт синхронизация атрибутов пользователя к учетным записям OneDrive

Следующим шагом является в файл запроса с помощью службы поддержки Майкрософт для синхронизации Azure Active Directory атрибут, выбранного на шаге 1, чтобы все учетные записи OneDrive в вашей организации. После этого синхронизации атрибута (и его значение), выбранные на шаге 1 будет сопоставлен с управляемым свойством, скрытые в SharePoint с именем `ComplianceAttribute`. Этот атрибут используется для создания фильтра поиска разрешения для OneDrive на шаге 4.
  
Отправьте этот запрос в службу технической поддержки Майкрософт содержат следующие сведения:
  
- По умолчанию имя домена вашей организации Office 365
    
- Имя атрибута Azure Active Directory (из шага 1)
    
- Ниже заголовок и описание назначения запроса поддержки: «Включить OneDrive для бизнеса с Azure Active Directory для соответствия требованиям безопасности фильтры синхронизации». Это позволит маршрутизации запроса для Office 365 обнаружения электронных данных команды разработчиков пользователей, которые будут реализованы запроса.
    
После разработки изменения и атрибута синхронизируется в OneDrive, службы поддержки Майкрософт будет отправлено номер сборки, который был изменен в и дату примерный развертывания. Обратите внимание на то, что процесс развертывания обычно занимает 4-6 недель после отправки запроса поддержки.
  
 **Важные:** Шаг 3-5 шаг можно выполнить перед развертыванием изменений. Однако выполнение контента поиск не возвращает документов из сайтов OneDrive, указанными в фильтре разрешения поиска до того времени, после развертывания изменения. 
  
## <a name="step-3-create-a-role-group-for-each-agency"></a>Шаг 3: Создание группы ролей для каждого бюро

Следующим шагом является создание группы ролей безопасности Office 365 &amp; центре соответствия требованиям, которая будет выровнена по вашей органов. Рекомендуется создать новую группу ролей, копирование группа руководителей встроенных eDiscovery, добавление соответствующих членов и удаление ролей, которые нельзя использовать в соответствии с потребностями. Дополнительные сведения о ролях, связанные с обнаружения электронных данных можно [назначать разрешения обнаружения электронных данных в Office 365 безопасность &amp; центре соответствия требованиям](assign-ediscovery-permissions.md).
  
Для создания группы ролей, перейдите на страницу **разрешений** безопасности &amp; центре соответствия требованиям и создания группы ролей для каждой группы в каждой бюро, которую будет использовать для управления расследований границы соответствия требованиям и вариантах eDiscovery. 
  
С помощью сценария границы Contoso соответствия требованиям, необходимо создать четыре группы ролей и соответствующие элементы добавлены в каждой из них.
  
- Руководители Fourth Coffee обнаружения электронных данных
    
- Fourth Coffee следствие
    
- Обнаружение электронных данных Coho Winery руководители
    
- Следствие Coho Winery
    

  
## <a name="step-4-create-a-search-permissions-filter-to-enforce-the-compliance-boundary"></a>Шаг 4: Создание фильтра поиска разрешений для применения параметров границ соответствия требованиям
<a name="step4"> </a>

После создания группы ролей для каждого бюро, следующим шагом является создание фильтры поиска разрешений, которые свяжите каждую группу ролей для его определенные бюро и определяет самой границе соответствия требованиям. Вам необходимо создать один фильтр поиска разрешения для каждого бюро. Дополнительные сведения о создании фильтров разрешения безопасности содержатся в разделе [разрешения Настройка фильтрации для поиска контента](permissions-filtering-for-content-search.md).
  
Ниже приведен синтаксис, который используется для создания фильтра поиска разрешения для соответствия требованиям границы.

```
New-ComplianceSecurityFilter -FilterName <name of filter> -Users <role groups> -Filters "Mailbox_<Compliance attribute from Step 1>  -eq '<AttributeVale> '", "Site_ComplianceAttribute  -eq <AttributeValue>' -or Site_Path -like <SharePointURL> *'" -Action <Action >
```
  
Ниже приведено описание каждого параметра в команде:
  
-  `FilterName`— Имя фильтра. Используйте имя, описание или идентифицирует бюро, позволяющие фильтровать будет использоваться в. 
    
-  `Users`— Задает пользователи или группы, получите этот фильтр для поиска контента действий, которые они выполняют. Для соответствия требованиям сети этот параметр указывает группы ролей (, созданной на шаге 3) в организацию, при создании фильтра для. Обратите внимание, что это является многозначным, поэтому можно включить один или несколько групп ролей, разделенных запятыми. 
    
-  `Filters`— Задает условия поиска для фильтра. Поиск границ соответствия требованиям необходимо определить следующие фильтры. Каждый из них относятся к расположение содержимого пользователя. 
    
  -  `Mailbox`— Задает почтовые ящики, группы ролей, определенные `Users` параметр может выполнять поиск. Для соответствия требованиям сети *ComplianceAttribute* — это тот же атрибут, который был определен на шаге 1 и *AttributeValue* указывает деятельностью организации. Этот фильтр позволяют участникам группы ролей только поиска почтовых ящиков в конкретных бюро; например `"Mailbox_Department -eq 'FourthCoffee'"` . 
    
  -  `Site`— Задает OneDrive учетные записи, группы ролей, определенные `Users` параметр может выполнять поиск. Для фильтра OneDrive используйте фактические строки `ComplianceAttribute`; Это будет сопоставляться с тем же атрибутом, который был определен на шаге 1, и, который синхронизируется учетным записям OneDrive в результате запроса поддержки отправки на этапе 2.  *AttributeValue* указывает деятельностью организации. Этот фильтр позволяют участникам группы ролей только поиска OneDrive учетные записи в конкретных бюро; например `"Site_ComplianceAttribute -eq 'FourthCoffee'"`.
    
  -  `Site_Path`— Задает сайтов SharePoint, которые определены группы ролей в `Users` параметр может выполнять поиск. *SharePointURL* указывает сайты в организацию, членов группы ролей могут выполнять поиск; Например`Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'"`
    
-  `Action`— Задает тип действия поиска соответствия, которая применяется фильтр. Например `-Action Search` будет применяться только фильтра при определении членов группы ролей в `Users` параметр выполняет поиск контента. В этом случае фильтр не применяется при экспорте результатов поиска. Соответствие требованиям сети, используйте `-Action All` , к которому применяется фильтр для всех действий поиска. 
    
    Список действий поиска контента см в разделе «New-ComplianceSecurityFilter» [Настройка разрешений фильтрации для поиска контента](permissions-filtering-for-content-search.md#new-compliancesecurityfilter).
    
Ниже приведены примеры фильтров разрешения два поиска, которые будут создаваться поддерживать сценарий, границы соответствия Contoso.
  
 **Fourth Coffee**

```
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_ComplianceAttribute -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL
```
   
 **Coho Winery**

```
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_ComplianceAttribute -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL
```

## <a name="step-5-create-an-ediscovery-case-for-an-intra-agency-investigations"></a>Шаг 5: Создание вариантом eDiscovery для исследования внутри бюро

Последним шагом является создание новое обращение eDiscovery в системы &amp; центре соответствия требованиям и добавьте группу ролей —, созданной на шаге 3 — как член регистр. Это приводит к два важных характеристик использования границы соответствия требованиям:
  
- Только для членов группы ролей, добавляемого в случае будет иметь возможность просматривать и получать доступ к case в системы &amp; центре соответствия требованиям. К примеру Если группа ролей Fourth Coffee следствие только членом дела, нажмите членов группы ролей руководители Fourth Coffee обнаружения электронных данных (или члены любой другой группе ролей) будет видеть или получить доступ к регистр.
    
- При выполнении поиска, связанных с обращением участника группы ролей, назначенных в случае они будут только возможность выполнять поиск расположения контента в пределах их бюро (который определяется разрешения поисковый фильтр, созданный на шаге 4.)


Чтобы создать новое обращение и назначьте члены:
    
1. Перейдите на страницу **обнаружения электронных данных** в системы &amp; центре соответствия требованиям и создайте новое обращение. 
    
2. В списке вариантах eDiscovery щелкните имя случая, которую вы только что создали.
    
3. На странице всплывающего **Управление в этом случае** в разделе **Управление группами ролей**, нажмите кнопку ![значком Добавить](media/8ee52980-254b-440b-99a2-18d068de62d3.gif) **Add**.
    
    ![Добавление группы ролей как члена вариантом eDiscovery](media/f8b4b557-01b9-4388-85be-b5b5ab7c5629.png)
  
4. В списке групп ролей выберите один из группы ролей, созданных в шаге 3 и нажмите кнопку **Добавить**.
    
5. Всплывающее окно **Управление в этом случае** для сохранения изменений нажмите кнопку **Сохранить** . 

## <a name="compliance-boundary-limitations"></a>Ограничения границ соответствия требованиям

Имейте в виду следующие ограничения, при управлении вариантах eDiscovery и расследований, использующие границ соответствия требованиям.
  
- При создании и выполнив поиск содержимого, можно выбрать контента расположения, находящиеся за пределами вашей бюро. Тем не менее из-за разрешений поисковый фильтр содержимого из этих папок не будут включены в результаты поиска.
    
- Границы соответствия требованиям, не применяются к удержаний в тех случаях, обнаружения электронных данных. Это означает, что диспетчер обнаружения электронных данных в одной бюро можно поместить пользователя в разных бюро на удержание. Тем не менее границы соответствия принудительно, если диспетчер обнаружения электронных данных выполняет поиск расположения контента пользователя, который был помещенные на удержание. Это означает диспетчер обнаружения электронных данных не будут иметь возможность поиска расположения контента пользователя, несмотря на то, что они смогли помещение пользователя на удержание.
    
    Кроме того удержаний статистики применяется только для размещения содержимого в деятельностью организации.
    
- Фильтры поиска разрешения не применяются к общие папки Exchange.

## <a name="searching-and-exporting-sharepoint-content-in-multi-geo-environments"></a>Поиск и экспорт содержимого SharePoint в средах с несколькими географически

Фильтры поиска разрешения также позволяют контролировать, где контент перенаправляется для экспорта, а какие центров обработки данных может быть выполнен поиск учетных записей OneDrive и сайты SharePoint в [среде SharePoint несколькими-географически](https://go.microsoft.com/fwlink/?linkid=860840):
  
- Экспорт результатов поиска в центре обработки данных. Это означает, что можно указать центра обработки данных расположения, что поиск, результаты будут экспортироваться из.
    
- Поиск маршрута сайтов SharePoint и учетных записях, OneDrive в центре обработки данных вспомогательных. Это означает, что можно указать расположение центра данных, где будет выполняться поиск.
    
Параметр **региона** для командлетов **New-ComplianceSecurityFilter** или **Set-ComplianceSecurityFilter** можно используйте для создания или изменения какие центра обработки данных экспорта будут направляться через.
  
|**Значение параметра**|**Расположение центра данных**|
|:-----|:-----|
|NAM  <br/> |Североамериканский (фактических данных, которые являются центры в США)  <br/> |
|EUR  <br/> |Европа  <br/> |
|APC  <br/> |Азиатско-Тихоокеанский регион  <br/> |
|CAN <br/> |Канада
   
Аналогично можно использовать следующие значения для значений параметров **области** для управления какой центре обработки данных, которая будет выполняться поиск в при поиске SharePoint и OneDrive расположения. Обратите внимание на то, что в следующей таблице показаны также какие экспортирует будут направляться через Центр обработки данных. 
  
|**Значение параметра**|**Центр обработки данных маршрутизации расположения для экспорта**|
|:-----|:-----|
|NAM  <br/> |RU  <br/> |
|EUR  <br/> |Европа  <br/> |
|APC  <br/> |Азиатско-Тихоокеанский регион  <br/> |
|CAN  <br/> |RU  <br/> |
|AUS  <br/> |Азиатско-Тихоокеанский регион  <br/> |
|KOR  <br/> |Центр обработки данных по умолчанию в организации  <br/> |
|GBR  <br/> |Европа  <br/> |
|JPN  <br/> |Азиатско-Тихоокеанский регион  <br/> |
|IND  <br/> |Азиатско-Тихоокеанский регион  <br/> |
|LAM  <br/> |RU  <br/> |
   
 **Примечание:** Если не указать параметр региона для фильтра разрешения для поиска, результаты поиска требуется экспортировать из ближайшего центра данных. 
  
Ниже приведены примеры использования **-региона** параметр во время создания фильтров разрешение поиска границ соответствия требованиям. Предполагается, что дочерним подразделением Fourth Coffee находится в Северной Америке и Coho Winery находится в Европе. 
  
```
New-ComplianceSecurityFilter -FilterName "Fourth Coffee Security Filter" -Users "Fourth Coffee eDiscovery Managers", "Fourth Coffee Investigators" -Filters "Mailbox_Department -eq 'FourthCoffee'", "Site_Department -eq 'FourthCoffee' -or Site_Path -like 'https://contoso.sharepoint.com/sites/FourthCoffee*'" -Action ALL -Region NAM
```

```
New-ComplianceSecurityFilter -FilterName "Coho Winery Security Filter" -Users "Coho Winery eDiscovery Managers", "Coho Winery Investigators" -Filters "Mailbox_Department -eq 'CohoWinery'", "Site_Department -eq 'CohoWinery' -or Site_Path -like 'https://contoso.sharepoint.com/sites/CohoWinery*'" -Action ALL -Region EUR
```
   
Имейте в виду при поиске и экспорт SharePoint и OneDrive следующее контента в средах с несколькими географически.
  
- Параметр **области** не управлять операций поиска почтовых ящиков Exchange; Поиск всех центров обработки данных будет выполняться при выполнении поиска почтовых ящиков. Для ограничения области поиска, из которых Exchange может быть выполнен почтовых ящиков, используйте параметр **фильтры** при создании или изменении фильтра разрешения для поиска. 
    
- Если это необходимо для обнаружения электронных данных диспетчера для поиска в нескольких регионах SharePoint, вам потребуется создать учетную запись другого пользователя, обнаружения электронных данных диспетчера, который может использоваться в фильтре разрешения поиска для указания альтернативных регион где Размещены сайты SharePoint или учетные записи OneDrive.
    
- Направляет при поиске контента в SharePoint и OneDrive, параметр **области** поиска на главном или вспомогательные расположения, в которых диспетчер обнаружения электронных данных будет проводить расследований обнаружения электронных данных. Если диспетчер обнаружения электронных данных выполняет поиск сайтов SharePoint и OneDrive вне области, которая указана в поисковый фильтр разрешения, результаты поиска будут возвращены. 
    
- При экспорте результатов поиска, контент из всех расположений контента (в том числе Exchange, Скайп для бизнеса, SharePoint, OneDrive и другие службы Office 365, которые можно выполнить поиск с помощью средства поиска контента) будет отправлен в расположение хранилища Azure в Центр обработки данных, который указан с помощью параметра **региона** . Это позволяет организациям соблюдения соответствия требованиям, не позволяя содержимого для экспорта в охраняемых границы. Если нет регион не указан в поисковый фильтр разрешения, содержимое загружается в организации область по умолчанию. 
    
- Можно изменить существующий фильтр поиска разрешения для добавления или изменения области, выполнив следующую команду:

    ```
    Set-ComplianceSecurityFilter -FilterName <Filter name>  -Region <Region>
    ```
 
## <a name="frequently-asked-questions"></a>Вопросы и ответы

 **Кто может создавать и управлять фильтры разрешения поиска (с помощью New-ComplianceSecurityFilter и командлеты Set-ComplianceSecurityFilter)?**
  
Создание, просмотр и изменение разрешений фильтры поиска, необходимо быть участником группы ролей управления организацией в системы &amp; центре соответствия требованиям.
  
 **Если диспетчер обнаружения электронных данных назначается несколько групп ролей, которая распределена среди нескольких агентств, как их поиск содержимого в одной агентство или другое?**
  
Диспетчер обнаружения электронных данных можно добавить параметры их поисковый запрос, который ограничивает поиск для конкретных бюро. Например если организация указала свойство **CustomAttribute10** для различения агентств, их можно добавить следующие их поисковый запрос для поиска почтовых ящиков и учетных записей OneDrive в конкретных бюро: `CustomAttribute10:<value> AND Site_ComplianceAttribute:<value>`.
  
 **Что происходит при изменении значения атрибута, который используется в качестве атрибута соответствия требованиям в поисковый фильтр разрешения?**
  
Необходимое до 3 дней для фильтра поиска разрешений для применения параметров границ соответствия при изменении значения атрибута, используемый в фильтре. Например в сценарии Contoso Предположим, что пользователь в бюро Fourth Coffee передается бюро Coho Winery. В результате значение атрибута **отдела** в объекте пользователя теперь *FourthCoffee* *CohoWinery* . В этом случае Fourth Coffee обнаружения электронных данных и инвесторов будут получать результаты поиска для этого пользователя для копирования 3 дней после изменения атрибута. Аналогичным образом он будет длиться до 3 дней до Coho Winery диспетчеры обнаружения электронных данных и следствие будут получать результаты поиска для пользователя. 
  
 **Видите диспетчер обнаружения электронных данных контента из двух отдельных соответствия границы**
  
Да. Это можно сделать, добавив пользователя в группы ролей, которые имеют возможность просмотра обоих агентств.
  
 **Поиск рабочих фильтры разрешения для обращения удержаний обнаружения электронных данных, политик хранения Office 365 или защиты от потери данных?**
  
Нет, не в настоящее время
  
 **Если указать регион для элемента управления, содержимое экспортируется, когда в этой области нет SharePoint организации, можно ли по-прежнему поиск SharePoint?**
  
Если область, указанную в фильтре разрешения поиска не существует в организации, поиск будет область по умолчанию.
  
 **Что такое максимальное число поисковых фильтров разрешения, которые могут быть созданы в организации?**
  
Нет ограничений на количество фильтров поиска разрешений, которые могут быть созданы в организации. Тем не менее производительность поиска будут затронуты при наличии более 100 поисковых фильтров разрешения. Для хранения количество фильтров разрешения поиска в вашей организации максимально, создайте фильтры, которые объединение правил для Exchange, SharePoint и OneDrive в фильтр разрешения единого поиска по возможности.
