# <a name="how-office-365-validates-the-from-address-to-prevent-phishing"></a>Как Office 365 проверяет адрес отправителя для предотвращения фишинга

Учетные записи электронной почты Office 365 и Outlook.com получают большое количество фишинговых атак. Одним из фишинговых фишингов является отправка сообщений, которые содержат значения для адреса From:, не соответствующие [RFC 5322](https://tools.ietf.org/html/rfc5322). Адрес "от:" также называется адресом 5322. from. Чтобы предотвратить такой тип фишинга, в Office 365 и Outlook.com необходимо, чтобы сообщения, полученные службой, включали в себя адрес, соответствующий спецификации RFC, как описано в этой статье.
  
> [!NOTE]
> Сведения, приведенные в этой статье, требуют базового представления общего формата адресов электронной почты. Для получения дополнительных сведений ознакомьтесь со статьей [rfc 5322](https://tools.ietf.org/html/rfc5322) (в частности, разделами 3.2.3, 3,4 и 3.4.1), [RFC 5321](https://tools.ietf.org/html/rfc5321), а также [RFC 3696](https://tools.ietf.org/html/rfc3696). В этой статье описывается применение политик для адреса 5322. from. Эта статья не относится к 5321. MailFrom Address. 
  
К сожалению, по-прежнему существуют некоторые устаревшие почтовые серверы в Интернете, которые продолжают отправлять "безопасные" сообщения электронной почты, отсутствующие или неправильно сформированные: адрес. Если вы регулярно получаете электронную почту от организаций, использующих эти устаревшие системы, рекомендуйте этим организациям обновлять свои почтовые серверы, чтобы соответствовать современным стандартам безопасности.
  
Корпорация Майкрософт начнет разбивать применение политик, описанных в этой статье, на 9 ноября 2017 г.

### <a name="all-messages-must-include-a-valid-from-address"></a>Все сообщения должны включать допустимый адрес от:
<a name="MustIncludeFromAddress"> </a>

Некоторые автоматические сообщения не включают адрес отправителя при отправке. В прошлое, когда Office 365 или Outlook.com получил сообщение без адреса "от:", служба добавила к сообщению следующий адрес по умолчанию: адрес, чтобы сделать его конечным.
  
```
From: <>
```

Начиная с 9 ноября 2017, Office 365 будет выполнять развертывание изменений в центрах обработки данных и почтовых серверах, которые приведут к применению нового правила, в котором сообщения без адреса "от:" не будут приниматься в Office 365 или Outlook.com. Вместо этого все сообщения, получаемые в Office 365, должны иметь действительный адрес от:. В противном случае сообщение будет отправлено в папки "Нежелательная почта" или "Удаленные" в Outlook.com и Office 365. 
  
### <a name="syntax-overview-valid-format-for-the-from-address-for-office-365"></a>Обзор синтаксиса: допустимый формат для адреса "от:" для Office 365
<a name="SyntaxOverviewFromAddress"> </a>

Формат значения адреса "от:" в нескольких документах RFC определен в подробностях. Существует множество вариантов адресации, которые могут считаться действительными или недопустимыми. Для простоты Корпорация Майкрософт рекомендует использовать следующий формат и определения:
  
```
From: "displayname " <emailaddress >
```

Где:
  
- Необязательно  *DisplayName* — это фраза, описывающая владельца адреса электронной почты. Например, это может быть более понятное имя, описывающее отправителя по сравнению с именем почтового ящика. Использование отображаемого имени не является обязательным. Тем не менее, если вы решили использовать отображаемое имя, корпорация Майкрософт рекомендует всегда заключать его в кавычки, как показано ниже. 
    
- Потребоваться  *EmailAddress* состоит из следующих компонентов: 
    
  ```
  local-part @domain
  ```

    Где:
    
  - Потребоваться  *Local-Part* — это строка, которая определяет почтовый ящик, связанный с адресом. Это значение уникально в пределах домена. Часто имя пользователя или GUID владельца почтового ящика используются в качестве значения для локальной части. 
    
  - Потребоваться  *domain* — полное доменное имя (FQDN) почтового сервера, на котором размещается почтовый ящик, определенный локальной частью адреса электронной почты. 
    
### <a name="format-of-the-from-address-if-you-dont-include-a-display-name"></a>Формат адреса "от:", если вы не укажете отображаемое имя
<a name="FormatNoDisplayName"> </a>

Правильно отформатированный адрес от: адрес, который не содержит отображаемого имени, включает только один адрес электронной почты с угловыми скобками или без них. Корпорация Майкрософт рекомендует не разделять угловые скобки пробелами. Кроме того, не включайте что-либо после адреса электронной почты.
  
Допустимы следующие примеры:
  
```
From: sender@contoso.com
```

```
From: <sender@contoso.com>
```

Приведенный ниже пример является допустимым, но не рекомендуемым, так как содержит пробелы между угловыми скобками и адресом электронной почты:
  
```
From: < sender@contoso.com >
```

Следующий пример является недопустимым, так как он содержит текст после адреса электронной почты:
  
```
From: "Office 365" <sender@contoso.com> (Sent by a process)
```

### <a name="format-of-the-from-address-if-you-include-a-display-name"></a>Формат адреса "от:", если вы включаете отображаемое имя
<a name="FormatDisplayName"> </a>

Для: адреса, включающие значение для отображаемого имени, применяются следующие правила.
  
- Если адрес отправителя включает отображаемое имя, а отображаемое имя содержит запятую, отображаемое имя необходимо заключить в кавычки. Пример:
    
    Приведенный ниже пример является допустимым:
    
  ```
  From: "Sender, Example" <sender.example@contoso.com>
  ```

    Следующий пример является недопустимым:
    
  ```
  From: Sender, Example <sender.example@contoso.com>
  ```

    Не заключайте отображаемое имя в кавычки, если это отображаемое имя включает запятую в соответствии со спецификацией RFC 5322.
    
    Рекомендуется заключить отображаемое имя в кавычки независимо от того, есть ли в отображаемом имени запятая.
    
- Если адрес отправителя включает отображаемое имя, адрес электронной почты должен быть заключен в угловые скобки.
    
    Рекомендуется вставлять пробел между отображаемым именем и адресом электронной почты.
    
### <a name="additional-examples-of-valid-and-invalid-from-addresses"></a>Дополнительные примеры допустимых и недопустимых адресов from:
<a name="Examples"> </a>

- Верно
    
  ```
  From: "Office 365" <sender@contoso.com>
  ```

- Недопустимый. Адрес электронной почты не заключен в угловые скобки:
    
  ```
  From: Office 365 sender@contoso.com
  ```

- Допустимый, но не рекомендуемый. Отображаемое имя не находится в кавычках. Рекомендуется заключить отображаемое имя в кавычки:
    
  ```
  From: Office 365 <sender@contoso.com>
  ```

- Недопустимый. Все заключено в кавычки, а не только отображаемое имя:
    
  ```
  From: "Office 365 <sender@contoso.com>"
  ```

- Недопустимый. Адреса электронной почты не заключены в угловые скобки:
    
  ```
  From: "Office 365 <sender@contoso.com>" sender@contoso.com
  ```

- Недопустимый. Между отображаемым именем и левой угловой скобкой нет пробелов:
    
  ```
  From: Office 365<sender@contoso.com>
  ```

- Недопустимый. Между закрывающей кавычкой и левой угловой скобкой нет пробела.
    
  ```
  From: "Office 365"<sender@contoso.com>
  ```

### <a name="suppress-auto-replies-to-your-custom-domain-without-breaking-the-from-policy"></a>Отмена автоматического ответа на личный домен без нарушения политики "от:"
<a name="SuppressAutoReply"> </a>

С помощью применения политики "создать из: политика" вы больше не можете использовать \< \> : для отключения автоответов. Вместо этого необходимо настроить пустую запись MX для личного домена.
  
Запись почтового обменника (MX) — это запись ресурса в DNS, которая определяет почтовый сервер, который получает почту для вашего домена. Автоматические ответы (и все ответы) подавляются, так как отсутствует опубликованный адрес, на который отвечающий сервер может отправлять сообщения.
  
При настройке пустой записи MX для личного домена:
  
- Выберите домен, из которого отправляются сообщения, которые не принимают (получают) сообщения электронной почты. Например, если ваш основной домен — contoso.com, вы можете выбрать noreply.contoso.com.
    
- Настройте пустую запись MX для вашего домена. Пустая запись MX состоит из одной точки, например:
    
  ```
  noreply.contoso.com IN MX .
  ```

Дополнительные сведения о публикации пустых MX приведены в статье [RFC 7505](https://tools.ietf.org/html/rfc7505).
  
### <a name="overriding-the-office-365-from-address-enforcement-policy"></a>Переопределение Office 365 from: политика принудительного применения адресов
<a name="Override"> </a>

После завершения развертывания новой политики можно обходить эту политику только для входящей почты, получаемой из Office 365, с помощью одного из следующих методов: 
  
- Списки разрешенных IP-адресов
    
- Правила для почтового процесса Exchange Online
    
Корпорация Майкрософт настоятельно рекомендует переопределять принудительное применение политики "от:". Переопределение этой политики может привести к снижению риска предоставления нежелательной почты, фишинга и других циберкримес в Организации.
  
Вы не можете переопределить эту политику для исходящей почты, отправляемой в Office 365. Кроме того, Outlook.com запрещает переопределение любого вида даже через поддержку.
