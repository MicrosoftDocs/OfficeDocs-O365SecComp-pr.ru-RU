# <a name="service-encryption-with-customer-key-for-office-365-faq"></a>Вопросы и ответы о службе шифрования с помощью ключа клиента для Office 365

В дополнение к базовой, уровень громкости шифрования, обеспечиваемой BitLocker и распределенного ключа Manager (DKM), Office 365 обеспечивает дополнительный уровень шифрования на уровне приложения для содержимого клиентов в Office 365, включая данные из Exchange Интернет-версия Скайп для бизнеса, SharePoint Online и OneDrive для бизнеса. Этот процесс называется службы шифрования.
  
Ключ клиента создан на основе службы шифрования и позволяет указать и ключи элемента управления, которые будут использоваться для шифрования данных на rest в Office 365, как описано в [Условия Online Services (OST)](https://www.microsoft.com/en-us/Licensing/product-licensing/products.aspx). Ключ клиента поможет вам провести обязательства соответствия требованиям, так как управлять ключами шифрования, которые использует Office 365 для расшифровки данных.
  
Чтобы оставить отзыв по ключу клиента, включая документацию, отправьте customerkeyfeedback@microsoft.com идеи, предложения и перспективы.
  
## <a name="what-is-service-encryption-with-customer-key"></a>Что такое служба шифрования с ключом клиента?

Ключ клиента — функция, которая позволяет вам потребуется подготовить и обслуживать клавиш, используемые для шифрования данных rest в Office 365. Компонент использует службы шифрования, которая является шифрования, который выполняется с Office 365 Exchange и SharePoint. Служба шифрования предоставляет преимущества за пределы какие BitLocker может предоставить — а именно, более высокой версии в глубокой от злоумышленников. Службы шифрования является строгое мера противодействия, если он пытается обхода системы управления доступа к Office 365, который будет использоваться для обработки всех запросов на доступ к данным клиента. Это, так как служба шифрования означает, что администратор сервера не имеет элемента управления или даже доступа для шифрования разделов и не может отключить шифрование, в отличие от с BitLocker. Таким образом злоумышленник с правами администратора на сервере размещения данных клиента, зашифрованные с помощью службы шифрования не сможет прочитать данные клиента и даже в том случае, если зашифрованные данные скопирован с сервера будет оставаться использовать невозможно. 
  
## <a name="what-office-365-data-at-rest-is-covered-by-customer-key"></a>Какие данные Office 365 статических покрывает ключа клиента?
<a name="WhatDataIsCoveredbyCustomerKey"> </a>

SharePoint Online контента сайта и файлов, хранящихся на этот сайт и файлы, загруженные в OneDrive для бизнеса рассматривается. Будет рассмотрен Exchange Online содержимого почтового ящика (тела сообщения электронной почты, элементы календаря и содержимого вложений электронной почты). Рассматривается текстовые беседы с Скайп для бизнеса, но вещание собрания Скайп записи и передача содержимого собрания Скайп не рассматриваются. Скайп распространения собрания и собрания Скайп контента передача шифруются вместе с другим содержимым в Office 365, но в настоящее время не предлагаются клиента управление ключами шифрования.

## <a name="what-is-the-availability-key"></a>Что такое ключ доступности?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

Ключ доступности является корневой раздел, который был подготовлен при создании политики шифрования данных. Клавиша доступности хранения и защищенные в Office 365 и функционально похож на двух корневые ключи, предоставленные пользователем, для использования с помощью службы шифрования с ключом клиента. В отличие от клавиш, которые предоставляют и управление ими в хранилище Azure ключ недоступны напрямую ключ доступности. Хранение и доступ к разделу доступности, намеренно отличаются от ключи помещение ключ Azure по трем причинам: во-первых, ключ доступности обеспечивает возможность высокой доступности в случае, если службы Office 365, не удается подключиться к ключи, размещенных в Azure Помещение; Во-вторых ключ доступность содержит функцию «приостановить стекло» в случае, если оба ключа помещение ключ Azure не сохраняются; и в-третьих, представлены в глубокой разделение логических элементов управления и обеспечивает защиту от потери всех ключей из одного атаки или точке сбоя. В конечном счете общего доступа к ответственности для защиты ключей, во время использования различных средства защиты и процессы, используемые для управления ключами снижает риск, что все разделы (и, следовательно, данные) будут утеряны или удален. Корпорация Майкрософт предоставляет единственного центра сертификации за уничтожения ключ доступности. В архитектуре никто в корпорации Майкрософт не имеет доступа к разделу доступности — доступно только для кода службы Office 365.
  
## <a name="how-many-data-encryption-policies-deps-can-i-create"></a>Сколько политики шифрования данных (DEPs) можно создать?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

 **Exchange Online и Скайп для бизнеса:** Можно создать до 50 DEPs. 
  
 **SharePoint Online и OneDrive для бизнеса:** Предотвращение выполнения данных относится к данных в одной географическому положению, также называемый географически. Если функция несколькими географически Office 365, можно создать одну DEP на географически. Если вы не используете несколькими географически, можно создать одну предотвращения выполнения данных.
  
## <a name="can-i-assign-a-data-encryption-policy-before-migrating-a-mailbox-to-the-cloud"></a>Можно назначить политику шифрования данных перед миграцией почтовых ящиков в облако?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

Да. Назначение политики шифрования данных (DEP) можно использовать командлет Windows PowerShell Set-MailUser для пользователя перед переносом почтового ящика в Office 365. При этом содержимое почтового ящика будет зашифрован с помощью назначенного DEP, как перенести содержимое. Это может быть более эффективным, чем назначение DEP после почтовый ящик уже были перенесены и затем ожидание шифрования вступили в силу, что может занять часа или возможно дней. 
  
## <a name="how-do-i-verify-that-encryption-with-customer-key-is-activated-and-office-365-has-finished-encrypting-with-customer-key"></a>Как убедитесь, что активации шифрования с ключом клиента и завершения шифрование с ключом клиента Office 365?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

 **Exchange Online и Скайп для бизнеса:** Вы можете [подключиться к Exchange Online с помощью удаленной оболочки PowerShell](https://technet.microsoft.com/en-us/library/jj984289%28v=exchg.160%29.aspx) и затем используйте командлет **[Get-MailboxStatistics]** для каждого почтового ящика, который необходимо вернуть. В выходные данные командлета Get-MailboxStatistics свойство _IsEncrypted_ возвращает значение **true** при шифровании почтового ящика и значение **false** , если это еще не. Если почтовый ящик зашифрован, значение, возвращаемое для свойства _DataEncryptionPolicyID_ — это GUID DEP, с помощью которого зашифрован почтового ящика. Дополнительные сведения о выполнении этого командлета [Get-MailboxStatistics](https://technet.microsoft.com/en-us/library/bb124612%28v=exchg.160%29.aspx) и с помощью PowerShell с Exchange Online см. 
  
Если почтовые ящики не шифруются после ожидания 72 часов с момента назначен DEP, инициировать перемещения почтовых ящиков. Для этого [подключения к Exchange Online с помощью удаленной оболочки PowerShell](https://technet.microsoft.com/en-us/library/jj984289%28v=exchg.160%29.aspx) и с помощью командлета New-MoveRequest и оставьте псевдоним почтового ящика следующим образом: 
  
```
New-MoveRequest <alias>
```

 **SharePoint Online и OneDrive для бизнеса:** Вы можете [подключиться к SharePoint Online PowerShell](https://technet.microsoft.com/en-us/library/fp161372.aspx)и затем используйте командлет **[Get-SPODataEncryptionPolicy]** для проверки состояния вашего клиента. ** _Состояние_** свойство возвращает значение **зарегистрирована** , если включено шифрование ключа клиента и зашифрованных все файлы всех веб-сайтов. Если по-прежнему в процессе выполнения шифрования, этот командлет предоставляются сведения о какой процент сайты завершена.

## <a name="how-is-the-availability-key-used-with-exchange-online"></a>Об использовании ключа доступности в Exchange Online?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

Существует три способа использования ключа доступности с Exchange Online:
  
- Служба доступности - в случае, если ключи помещение ключ Azure, отсутствует.
    
- Действия запускаются кодом службы Office 365 — например, создание индекса поиска или перемещение почтовых ящиков.
    
- Восстановление из ключевые потери — например, выхода оба ключа помещение ключ Azure, связанные с одной функции.
    
 **С помощью ключа доступности службы доступности в случае ключи помещение ключ Azure, отсутствует.**
  
Office 365 использует параметр доступности для службы доступности и восстановления в недопустимом состоянии ключа клиента для Exchange Online. Существует иерархии ключей, используемых с ключом клиента. На следующем рисунке показан этой иерархии.
  
![](media/a760156b-737f-469a-80ab-c28b7a8b9160.png)
  
Если оба ключа помещение ключ Azure из одного данных шифрования политики (DEP) недоступны, Office 365 можно использовать доступность ключ для изменения в новый Office 365 функции. определяет, следует ли использовать ключ доступности службы доступности по-разному в зависимости от операции, инициированные пользователем, например, когда пользователь загружает электронной почты в клиенте Outlook или действия инициировал системы, такие как индексации содержимого почтового ящика или поиска обнаружения электронных данных запускается процесс.
  
Office 365 следующим образом этот процесс в ответ на инициированных пользователем действия, чтобы определить, следует ли использовать ключ доступность для почтовых ящиков пользователей:
  
1. Office 365 считывает DEP, назначенных почтовому ящику для определения местоположения разделов два клиента в хранилище Azure ключ.
    
2. Office 365 случайным образом выбирает один из двух клиента разделов из DEP и отправляет запрос хранилище ключа Azure развертывать ключ DEP, с помощью ключа клиента.
    
3. Если запрос развертывать DEP ключевые с помощью ключа клиента, возникает ошибка и возвращает ошибку, Office 365 отправляет второй запрос хранилище ключа Azure, данный момент его на использование общей папки с инструкциями (второй) ключа клиента.
    
4. Если второй запрос развертывать DEP ключа, с помощью ключа возникает ошибка клиента и возвращает ошибку, Office 365 проверяет результаты из обоих запросов:
    
  - Если анализ определяет, что ошибок не отражают явно с идентификатором клиента, Office 365 использует параметр доступности для расшифровки DEP ключа. Предотвращение выполнения данных затем используется для расшифровки ключа почтовых ящиков и завершения запроса пользователя.
    
    В этом случае хранилище Azure ключ недоступен или не может ответить по любой причине. Office 365 не имеет возможности определения, если клиент специально был отозван доступ к разделам.
    
  - Если изучение указывает, выбирайте действие выполнено для визуализации клиентом ключи недоступен, а затем ключ доступности не будет использоваться, не удается выполнить запрос пользователя и пользователь получает сообщение об ошибке, таких как ошибка входа в систему.
    
    В этом случае выполняется принять во внимание, на которые оказывает службы, что условие ключа клиента неработоспособные клиента. К примеру Если клиент использует один DEP для всех почтовых ящиков в организации, клиента могут возникать широко распространены сбоя, где пользователи не могут получить доступ к почтовым ящикам. Это гарантирует, что при неработоспособные оба ключа клиента клиента становится известно о необходимости исправить ситуацию и восстановление службы в работоспособном состоянии.
    
 **С помощью ключа доступности для действий, инициированных кода службы Office 365.**
  
Кода для Office 365 службы всегда имеет маркер допустимое имя входа и не может быть заблокирован. Таким образом пока не будет удалено ключ доступности, его можно использовать для действия запускаются, или внутри кода службы Office 365, такие как создание индекса поиска или перемещение почтовых ящиков.
  
 **С помощью ключа доступности для восстановления из потери ключей.**
  
Ключ доступности можно использовать для восстановления из потери оба ключа помещение ключ Azure, связанных с одной DEP, как описано в ответ на часто задаваемые вопросы о записи «Если Мои ключи удаляются, как освободить?».
  
## <a name="how-is-the-availability-key-used-with-sharepoint-online-and-onedrive-for-business"></a>Как ключ доступности используется с SharePoint Online и OneDrive для бизнеса?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

SharePoint Online и архитектура OneDrive для бизнеса и внедрения для ключа клиента и доступность ключа отличаются от Exchange Online и Скайп для бизнеса.
  
При клиент перемещается на управляемых клиентов ключи, Office 365 создает ключ промежуточных конкретных клиентов (TIK). Office 365 шифрует TIK дважды, один раз с каждым из ключей клиентов и сохраняет зашифрованные две версии TIK. Хранятся только зашифрованные версии TIK и TIK могут быть расшифрованы только с ключами клиента. TIK затем используется для шифрования ключи сайта, которые будут использоваться для шифрования ключей больших двоичных объектов. Большие двоичные объекты сами шифруются и хранятся в службе хранилища больших двоичных объектов Microsoft Azure.
  
Office 365 следующим образом эту процедуру, чтобы получить доступ к BLOB-объект, который содержит данные файла клиента:
  
1. Расшифровки TIK, с помощью ключа клиента.
    
2. Используйте расшифрованного TIK для расшифровки ключа узла.
    
3. Используйте ключ расшифрованного сайта для расшифровки ключа больших двоичных объектов.
    
4. Используйте ключ расшифрованного больших двоичных объектов для расшифровки.
    
При расшифровке TIK Office 365 вопросы два запроса расшифровки хранилище ключа Azure с небольшим смещением. Первый завершиться предоставляющей результатов, Отмена других запросов.
  
В случае, если клиент теряет доступа к ключам клиента, Office 365 также шифрует TIK ключом доступности и сохраняет это вместе с TIKs, зашифрованные с помощью ключа каждого клиента. TIK, зашифрованные с помощью ключа доступности используется только в том случае, если клиент вызывает Microsoft для прикрепления пути восстановления, когда они теряют доступ к ключам, намеренно или случайно.
  
Для обеспечения доступности и для обеспечения масштаба расшифрованного TIKs кэшируются в кэш-памяти временными. Два часа до кэша TIK срок действия которых истекает, пытается расшифровать каждого TIK Office 365. Расшифровка TIKs расширяет время жизни кэша. В случае сбоя для значительное время расшифровки TIK Office 365 создает оповещения для уведомления engineering до истечения срока действия кэша. Только в том случае, если клиент вызывает Microsoft Office 365 будет инициировать операцию восстановления, которая включает в себя расшифровывать TIK с ключом доступности хранятся в корпорации Майкрософт секретный хранилища и входящая клиента еще раз с помощью расшифрованного TIK и новый набор ключи обеспечивается хранилище ключа Azure.
  
Сегодня ключа клиента участвует в цепочке шифрования и расшифровки SharePoint Online файлов данных, сохраненных в хранилище Azure больших двоичных объектов, но не SharePoint Online элементов списка или метаданные, которые хранятся в базе данных SQL. Office 365 доступности раздел используется для SharePoint Online или OneDrive для бизнеса отличный от случая, указанных выше, являющийся Клиент инициировал. Человеческого доступ к данным клиента был защищен с защищенного хранилища клиента.
  
## <a name="how-is-customer-key-licensed"></a>Как лицензируется ключа клиента?
<a name="DiffCustomerKeyandBYOKAzureIP"> </a>

Ключ клиента предлагается в набор Office 365 корпоративный, «E5» и SKU дополнительные соответствия требованиям. Кроме того необходимо также покупают клиенты соответствующей лицензии для использования ключа хранилище Azure.
  
Каждому пользователю выгодное ключа клиента должен иметь лицензию, если необходимо быть, к которому применяется ключа клиента.
  
Для SharePoint Online администратор Office 365, который настраивает ключа клиента также необходимо иметь лицензию для выполнения действий программы установки. Кроме того пользователи, которые выгодное компонента должны иметь лицензию - включает владельца веб-сайта и всех пользователей, доступа к файлам на одном или нескольких сайтах, зашифрованные с помощью ключа клиента. Внешние пользователи не требуется иметь лицензию для доступа к файлам на одном или нескольких сайтах, зашифрованные с помощью ключа клиента.
  
Для Exchange Online необходимо иметь лицензию почтовые ящики «user» и «пользователя почты». Все остальные, таких как общие почтовые ящики не требуется иметь лицензию для ключа клиента. Чтобы проверить, что ваш почтовый ящик Exchange Online действующую лицензию, выполните следующий командлет:
  
```
(Get-Mailbox <alias >).PersistedCapabilities
```

Если существует строка BPOS_S_EquivioAnalytics почтовый ящик должным образом лицензию. В противном случае следует применить нужной лицензии перед использованием функции ключа клиента для этого почтового ящика.
