Вы можете использовать широкий спектр файлов журналов веб-трафика и источников данных для Office 365 Cloud App Security. Тем не менее, файлы журнала веб-трафика должны содержать определенную информацию и иметь определенный способ форматирования, чтобы они работали с отчетами об обнаружении приложений Office 365 Cloud App Security и информационной панели облачного обнаружения. Используйте эту статью в качестве справочного руководства по журналам веб-трафика и источникам данных, которые вы будете использовать в Office 365 Cloud App Security.

Ознакомьтесь с разРешениями [в центре &amp; безопасности и соответствия требованиям Office 365](permissions-in-the-security-and-compliance-center.md). 
  
## <a name="web-traffic-log-requirements"></a>Требования к журналу веб-трафика

Office 365 Cloud App Security использует данные в журналах веб-трафика, чтобы определить, какие приложения используются в Организации. Дополнительные сведения, которые включены в файлы журналов, лучше отображаются в действиях пользователей.
  
В следующих разделах перечислены необходимые атрибуты и дополнительные требования к журналам веб-трафика для корректной работы в Office 365 Cloud App Security.

### <a name="attributes"></a>Атрибуты

Приложение Office 365 Cloud App Security не может отображать или анализировать атрибуты, которые не включены в журналы веб-трафика. Например, стандартный формат журнала брандмауэра Cisco ASA не содержит количество отправленных байтов на транзакцию, имя пользователя или целевой URL-адрес (только конечный IP-адрес). Таким образом, эти атрибуты не отображаются в данных обнаружения в облаке, и их видимость в облачных приложениях ограничена. Для брандмауэров Cisco ASA уровень информации должен быть равен 6. 

Журналы веб-трафика должны включать следующие атрибуты:

- Дата операции
- Исходный IP-адрес
- Пользователь-источник (настоятельно рекомендуется)
- Конечный IP-адрес
- URL-адрес назначения (рекомендуется; URL-адреса обеспечивают более высокую точность обнаружения облачного приложения по сравнению с IP-адресами.
- Общий объем данных (рекомендуется, по крайней мере, информация о данных очень ценная)
- Количество отправленных или загруженных данных (рекомендуется, содержит сведения о шаблонах использования облачных приложений)
- Выполняемое действие (разрешено или заблокировано)

### <a name="additional-requirements"></a>Дополнительные требования 

В дополнение к включению атрибутов, перечисленных ранее в этой статье, журналы веб-трафика должны удовлетворять следующим требованиям:

- Необходимо поддерживать источник данных для файлов журналов.
- Формат, используемый файлами журнала, должен быть соответствующим стандартному формату. Когда файл будет отправлен, при обнаружении приложения оно будет проверено.
- События в журнале должны быть выполнены не более 90 дней назад.
- Файл журнала должен включать данные исходящего трафика, которые можно проанализировать для сетевой активности.

- Клавистер НГФВ (syslog)

- Фортинет Фортигате
- Безопасный облачный шлюз ибосс
- Шаблоны Juniper СРКС
- Шаблоны Juniper ССГ
- Безопасный веб-шлюз McAfee
- Microsoft Forefront Threat Management Gateway (W3C)

> [!NOTE]
> Если источник данных, который вы хотите использовать, отсутствует в этой статье, вы можете запросить его добавление в Обнаружение приложений. Для этого при создании отчета выберите вариант **другой** для **источника данных**. Затем введите имя источника данных, который вы пытаетесь отправить. Мы проверим журнал и узнаете, можно ли добавить поддержку для этого типа журнала. Кроме того, вы можете [определить настраиваемое средство синтаксического анализа](https://docs.microsoft.com/cloud-app-security/custom-log-parser) , которое соответствует вашему формату. 
  
## <a name="troubleshoot-errors-when-log-files-are-uploaded"></a>Устранение ошибок при загрузке файлов журнала

После загрузки файлов журнала веб-трафика просмотрите журнал управления, чтобы узнать, возникли ли какие-либо ошибки. При возникновении ошибок используйте сведения из следующей таблицы, чтобы устранить эти ошибки.
  
|**Ошибка**|**Описание**|**Решение**|
|:-----|:-----|:-----|
|Неподдерживаемый тип файла  <br/> |Переданный файл не является допустимым файлом журнала. Например, файл изображения.  <br/> |Отправьте текстовый, ZIP-или файл gzip, который был напрямую экспортирован из брандмауэра или прокси-сервера.  <br/> |
|Внутренняя ошибка  <br/> |Обнаружен внутренний отказ ресурса.  <br/> |Нажмите кнопку **повторить** , чтобы повторно запустить задачу.  <br/> |
|Формат журнала не отвечает  <br/> |Отправленный формат журнала не соответствует ожидаемому формату журнала для этого источника данных.  <br/> |
Убедитесь, что журнал не поврежден. Сравните и сопоставьте формат файла журнала с форматом примера, показанным на странице отправки. |
|Количество транзакций, которые старше 90 дней  <br/> |Все транзакции старше 90 дней, поэтому они игнорируются.  <br/> |Экспорт нового журнала с последними событиями и повторное его отправка.  <br/> |
|Нет транзакций для каталог облачных приложений  <br/> |В журнале не будут найдены транзакции для любых распознаваемых облачных приложений.  <br/> |Убедитесь, что журнал содержит данные исходящего трафика.  <br/> |
|Неподдерживаемый тип журнала  <br/> |При выборе параметра **источник данных = другой (неподдерживаемый)** журнал не анализируется. Вместо этого он отправляется на проверку в службу технической поддержки [Microsoft Cloud App Security](https://aka.ms/whatiscas) .<br/> |Техническая группа [Microsoft Cloud App Security](https://aka.ms/whatiscas) выполняет построение выделенного анализатора для каждого источника данных. Большинство популярных источников данных уже поддерживаются. При отправке неподдерживаемого источника данных он просматривается и добавляется в список возможных новых средств синтаксического анализа источника данных.<br/> Когда к компоненту добавляется новый анализатор, уведомление о выпуске Microsoft Cloud App Security включается в заметки о выпуске Microsoft Cloud App Security.  <br/> |
